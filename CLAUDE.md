## プロジェクト概要

- **kuroji（黒字）**: 貸借対照表(BS)・損益計算書(PL) など**会計の仕組みとルールを学びながら会社を経営する**ターン制シミュレーションゲーム（公開 Web アプリ）。
- **目的**: プレイヤーの経営判断（設備投資・仕入・価格設定・人員・資金調達 等）が**財務諸表にどう反映されるか**を体験し、会計恒等式や主要な経営指標を理解しながら、**純資産を増やす（黒字経営を続け倒産を避ける）戦略**を磨く。
- **学習主眼**: ゲームとして遊べることよりも、**会計構造（三表のつながり）と経営戦略の理解**を最優先する。
- **非目標**: 実在企業の財務再現や会計実務ツールではない。あくまで学習用の簡略モデル。**個人情報は扱わない**（全てブラウザ内計算、サーバー送信なし）。
- **公開前提**: 静的サイトとして GitHub Pages に公開する。

## 必須プロセス（最重要）

1. **TDD（テスト駆動）**: 会計ロジックは Vitest で Red→Green→Refactor。とくに**会計恒等式（資産 = 負債 + 純資産）**、当期純利益→利益剰余金の流れ、CF と現金増減の整合は、実装より先にテストで固定する。`core/` の純粋関数を中心に検証する。
2. **会計パラメータ（税率・各種定数）は data として分離**: 法人税の実効税率・減価償却率・市況パラメータなどは `data/` に**ロジックから分離**して置く。数値だけ差し替え・検証できる構造にし、根拠のある数値には**出典をコメントで残す**。
3. **決定論を保つ**: ランダム要素は**シード付き RNG**で扱い、`Math.random()` を計算ロジックに直接使わない。同じ入力＋シードで同じ結果になること（リプレイ・テスト可能性）を守る。
4. **個人情報・シークレットを絶対にコミットしない**。本アプリは原則持たないが `.env*` は `.gitignore` 済み。

## コマンド

```bash
npm install        # 依存インストール
npm run dev        # ローカル開発サーバー（Vite, http://localhost:5173）
npm run test       # Vitest（会計ロジックのテスト、1回実行）
npm run test:watch # Vitest（watch モード）
npm run typecheck  # 型チェックのみ（tsc -b --noEmit）
npm run build      # 本番ビルド（tsc -b → vite build → dist/）
npm run preview    # 本番ビルドのローカルプレビュー
```

公開: `main` への push で GitHub Actions（`.github/workflows/deploy.yml`）が
テスト→ビルド→GitHub Pages へ自動デプロイする。本番ビルドのベースパスは
`vite.config.ts` で `/kuroji/`（リポジトリ名）に設定。

## アーキテクチャ早見表

- `core/` … UI 非依存の**ゲーム/会計ドメイン層**（純粋関数）。ターン解決（`engine/resolveTurn.ts`）、BS/PL/CF の生成、減価償却・利息・税・経営指標、需要モデル（`market/demand.ts`）、原材料スポット価格（`market/material.ts`）、研究開発による製品パラメータ（`product/research.ts`）、シード付きRNGとイベント（`util/rng.ts`・`engine/events.ts`）。**棚卸資産は原材料・製品に分かれ、各々「数量」と「移動平均評価額」の二面で管理**（仕入→生産で原材料を製品へ価値保存の振替→販売）。原価が変動しても恒等式を保つ。Vitest でテストする中心。`data/` のパラメータを**注入されて**計算する。
- `data/` … シナリオ別の初期条件・市況パラメータ・イベント・会計定数（出典コメント付き）。`types.ts`（集約型）＋ `<シナリオ>.ts`（定数）＋ `index.ts`（レジストリ `getScenario`/`AVAILABLE_SCENARIOS`）の構成。
- `app/` … React の UI（財務諸表ダッシュボード・経営判断パネル・指標表示）。`core/` を呼ぶだけにし、計算は UI に書かない。

## 設計上の約束

- **計算は `core/` の純粋関数**に集約し、UI（`app/`）から完全分離する。
- **会計の整合性を最優先**: まず「状態遷移＋恒等式アサート」で複式の整合を担保し、必要に応じて仕訳レベルへ拡張する。
- **拡張容易性**: 市況・難易度・イベントは `data/` のシナリオを差し替えるだけで変えられる設計にする。
- UI はすべて**日本語**。会計用語には学習者向けの説明（ツールチップ等）を添える。
- Secrets / バックエンド無しで動く（完全クライアントサイド）。
