## 1. 商材開発（Product Development）

### ねらい（kuroji の核＝会計学習に効く点）
既存の R&D（`rdStock`→原価↓/需要↑の連続投資・**費用処理**）を一歩進め、
**開発投資の資産計上 → 無形固定資産 → 償却** を体験させる。
「研究開発費は費用、でも製品化投資（ソフトウェア等）は資産になる」という、会計で最も誤解されやすい論点を三表で学ぶ。
**のれん（無形資産の毎期償却）**の機構をそのまま再利用でき、企業ライフサイクルの自然な次の層。

### コア機構（開発プロジェクト方式・製造業/IT）
1. **企画**: 「新商材を開発」でプロジェクト開始（種別＝新規 or 改良、対象、予算を設定）
2. **開発中**: 毎期 開発費を投じる。**費用にならず「開発仮勘定」という資産に積み上がる**（現金↓・資産↑＝投資CF・P/L無傷）
3. **完成・ローンチ**: 累計投資が必要額に達すると、開発仮勘定を**「無形固定資産」へ振替**し、
   **新しい製品ラインとして販売開始**（新規）／**既存ラインの rd 係数を一段強化**（改良）
4. **回収**: 無形資産を製品寿命にわたり**毎期償却**（販管費・のれん償却と同型）。新商材の売上−償却費で投資を回収
5. **陳腐化**: ローンチ後、需要が逓減 → 次の開発を促す（ポートフォリオ管理の学び）

これで「先行投資（現金が出て利益は出ない）→資産が育つ→ローンチ→売上＋償却の重し」という
**開発のJカーブ**を三表上で追体験できる。恒等式は全段階で厳密維持。

### シナリオ別テーマ（作るものを変える／ユーザー要望・確定）
| シナリオ | 開発するもの | 無形資産の名称 | 会計処理 | 陳腐化（lifecycle） |
|---|---|---|---|---|
| 標準（製造業） | 新製品／製品改良 | 製品開発資産 | 資産計上→償却 | `decay`（じわ減り・中庸） |
| IT（startup） | 新機能・新サービス | ソフトウェア | 資産計上→償却 | `decay`（速い・技術陳腐化） |
| カフェ | 定番メニュー／季節限定メニュー | （資産化しない） | **費用処理** | 定番=`permanent`／季節=`seasonal` |

### 陳腐化の3表現（lifecycle・一般化）
- `decay`: ローンチ後、需要が毎期一定率で逓減（製造業/IT。次の開発を促す）
- `permanent`: 逓減なし・恒久ブースト（カフェ定番メニュー）
- `seasonal`: `boostDuration` 期間だけ需要を押し上げ、期間終了で効果消滅（カフェ季節限定メニュー＝旬のメニュー）

### カフェの扱い（確定・「1と2の両立」）
メニューはレシピで資産化しない現実に合わせ、**どちらも費用処理（「メニュー開発費」＝販管費・即時）**。2種類を開発できる:
- **定番メニュー**: 恒久ブースト（`permanent`）。長く売れる看板メニュー。
- **季節限定メニュー**: 一時ブースト（`seasonal`・期間限定で終了）。旬・トレンドの押し上げ。
狙い: 製造業/IT の**資産計上→償却**と、カフェの**費用処理（人件費・レシピは簿外）**を対比させ、
「開発には資産化されるものと費用処理されるものがある」を業種差そのもので教える。
季節限定は「計画された短命」＝じわ減りの陳腐化とは別の“終わり方”を体験させる。

### 新規 vs 改良（両方選べる／ユーザー要望）
- **新規**: 開発完了で新しい `productLine` が実行時に増える（ポートフォリオ拡大）
- **改良**: 既存ラインの rd 係数（原価減/需要増の上限）を一段引き上げ（v1→v2→…）

### データ / 状態（骨子）
- `SimParams.devProjects: DevProject[]`（id・name・kind:'new'|'upgrade'・対象（新ライン定義 or 既存lineId）・
  requiredInvestment・minTurns・**capitalize:boolean**（製造/IT=true・カフェ=false）・amortRate・
  **lifecycle:'decay'|'permanent'|'seasonal'**・obsolescenceRate（decay時）・boostDuration（seasonal時））
- `CompanyState.devInProgress: {projectId, invested, startedTurn}[]` ／ `developedLines: ProductLineState拡張` ／
  ローンチ済みラインの `launchedTurn`（陳腐化計算用）
- `BalanceSheet.fixedAssets.developmentAsset?`（開発仮勘定＋無形資産の簿価。のれんの隣の無形資産）
- `Decision.devSpend?: Record<projectId, number>` ／ `startDev?: projectId` ／ （前倒しローンチは要検討）
- `TurnResult`: devAmortized・launchedThisTurn 等（表示・診断用）

### 会計の恒等式（全段階で維持・テストで固定）
- 資産計上: cash↓ ＝ 開発仮勘定↑（**資産内の付け替え**・投資CF・equity 不変）
- 振替（ローンチ）: 開発仮勘定 → 無形資産（**資産内振替**・恒等式不変）
- 償却: 無形資産↓・販管費↑ → 純利益↓ → 利益剰余金↓（のれん同型・営業CFに足し戻し）
- 費用処理（カフェ）: 開発費 → 販管費（即・資産計上なし）

### UI（事業タブに「商材開発」パネル）
- 進行中案件: 進行バー・累計投資・ETA・完成予定効果／「新商材を開発」ボタン（種別・対象・予算）
- 完成品は判断パネルに新ラインとして出現（改良は既存ラインにバッジ）
- B/S に「開発資産（仕掛＋無形）」行・用語集に「開発費の資産計上」「無形固定資産」「償却」

### テスト（TDD 先行）
資産計上で恒等式/投資CF/P/L無傷・ローンチ振替で恒等式不変＋新ライン出現・償却でopEx↑＆CF足し戻し・
陳腐化で需要逓減（決定論）／カフェは逓減なし・費用処理シナリオで販管費計上＆資産計上なし・
恒久不変条件スイートに devProjects を含めて反例ゼロ。SAVE_VERSION は optional 追加なら据置（移行基盤で整合検査）。

## 2. 人材開発（HR Development）— 本格版（個人・役割・経験）

### ねらい（商材開発と“対”の会計フック）
商材開発が「**資産計上**」を教えるのに対し、人材開発は**「人的資本は簿外（B/Sに載らない）」**を教える。
研修費・人件費はすべて**費用処理**。「人は最大の資産」と言われるのに貸借対照表には出ない、という会計の非対称を体験。
参考メトリクスとして「**人的資本（簿外・推定値）**」を可視化し、"価値はあるがB/Sに出ない"を明示する。

### モデル（従業員を個人単位で保持）
現状の `headcount`×`laborPerHead` スカラーを、**従業員の配列**に置き換える（シナリオ opt-in・既存スカラーは後方互換で残す）。
```
Employee = {
  id
  role: 'field' | 'mgmt' | 'rnd'   // シナリオでラベルが変わる（下表）
  grade: 1..maxGrade               // 等級（昇進で上がる）
  exp: number                      // 経験（在籍＋研修で蓄積→スキル・昇進資格）
  morale: 0..1                     // 士気
}
```
- **スキル** = f(grade, exp)（逓減）。**生産性** = laborPerHead × スキル × 士気係数。
- **役割**:
  - `field`（現場）: 労働能力（生産）を供給。
  - `mgmt`（管理）: 直接は作らないが、全社の生産性・士気に補正（チーム効率）＋不良/離職を抑制。
  - `rnd`（研究）: 毎期 rdStock に受動的に寄与（商材開発とシナジー）。

### 4つのレバー
1. **採用**: junior（grade1・exp0・低スキル低賃金）を追加。**急な大量採用は平均スキルを薄める**（＝人を増やせば即戦力ではない）。
2. **研修投資（費用）**: `Decision.trainingSpend` → 全社/対象の exp・スキル↑（逓減）。**昇進を早め**、品質・定着も上げる。
3. **昇進（昇給）**: exp が等級しきい値に達すると昇進資格。昇進で **grade↑ → 賃金↑（昇給・人件費↑）＋生産性↑**。
   自動昇進（資格到達で）＋ UI で確認、を基本に。「育てると強くなるが人件費も上がる」ROI 判断を学ぶ。
4. **給与・処遇**: `wageLevel`（相場比）＋等級別賃金。相場割れ・低士気で離職。

### 士気・過重労働（採用・確定）
- 士気の増減要因: 給与（相場比）・**過重労働**（生産が能力を超える状態が続くと疲弊）・研修（人への投資）・昇進。
- 低士気 → **離職率↑・生産性↓**。とくに高等級の離職は痛い（生産性喪失＋再採用は junior から）。
- 「人を酷使すると崩れる／育てると報われる」を体験。研修・適正人員・昇進で回復。

### 会計（すべて費用処理・簿外）
- 賃金（等級別合計）・研修費・採用費・退職金・昇給 → **すべて P/L の人件費/研修費（即費用）**。
- **人的資本は B/S に計上しない**（簿外）。参考メトリクス「人的資本（簿外・推定）」で可視化。
- 商材開発（資産化→償却）との**対比が学びの核**。用語集に「人的資本」「研修費（費用処理）」「簿外」。

### シナリオ別（役割ラベル・賃金・傾向）
| シナリオ | 役割ラベル | 特徴 |
|---|---|---|
| 標準（製造業） | 職人／班長／技術者 | 熟練で能力↑・不良↓。定着はやや良い |
| IT（startup） | エンジニア／リード／PM | スキルで能力↑＋rnd 貢献。流動性高く高給・離職しやすい |
| カフェ | スタッフ／店長／シェフ | 回転率↑・接客品質↑。若年で離職しやすい・研修効果は早い |

### データ / 状態
- `CompanyState.employees?: Employee[]`（headcount は `employees.length` の導出値に）。opt-in（未指定シナリオは従来スカラー）。
- `SimParams.roles?` / `grades?`（等級別: 賃金倍率・生産性倍率・昇進 exp しきい値）/ `trainingParams`（効き・逓減）/
  `moraleParams`（過重労働しきい値・回復/悪化率・離職への効き）。
- `Decision.trainingSpend?` ／ `hire` は role 指定に拡張 ／ 昇進は自動＋確認（`promoteConfirm?`）。
- `TurnResult`: 平均スキル・士気・昇進数・離職の内訳・人的資本推定 等（表示・診断用）。

### 会計の恒等式（維持）
人件費・研修費・昇給・退職金はすべて P/L 費用 → 純利益↓ → 利益剰余金↓（現金は当期支払分だけ流出）。
B/S に人的資産を作らない＝恒等式の左辺は不変。既存の労働ブロックの置換のみで、棚卸/固定資産/財務には触れない。

### UI（事業タブに「人材・組織」パネル）
- 従業員リスト（役割・等級・スキル・士気のバッジ）／平均スキル・士気ゲージ／過重労働アラート。
- 研修費入力・採用（役割選択）・昇進確認。人的資本（簿外・参考値）メトリクス。

### 規模と段取り（★最大の機能）
全機能で最重。スカラー労働 → 個人配列への置換は resolveTurn の労働ブロック全面改修。**フェーズ分割＋重TDD**推奨:
- HR-0: Employee 配列導入（opt-in）・生産性=Σ個人・恒等式の回帰（既存スカラー挙動と一致）
- HR-1: 研修投資（費用）→スキル→生産性/品質
- HR-2: 経験蓄積・昇進・昇給（等級）
- HR-3: 士気・過重労働・離職の刷新
- HR-4: 役割（mgmt/rnd）・シナリオ別・バランス再計測
恒久不変条件スイートに employees を含め反例ゼロ。SAVE_VERSION は構造変更のため +1＋移行（スカラー→employees 生成）。

## 実装の順番（設計フェーズの結論）
1. **商材開発**（中〜重・のれん機構を再利用・資産計上/償却/陳腐化3表現）
